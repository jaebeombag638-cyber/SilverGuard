# SilverGuard 프로젝트 계획

## 목표


- 어르신(실버세대)이 받은 의심스러운 문자를 입력하면, 위험도를 판단하고 왜 위험한지 + 어떻게 대응해야 하는지 가이드를 제공하는 데모 앱
- 핵심은 "모델 성능"을 보여주는 것 — 단순 LLM API 프롬프팅에 그치지 않고 실제 파인튜닝 결과물을 포함

## 범위

### 포함

- **URL 위험도 판별기 (파인튜닝 모델)**
  - 입력: 문자 메시지 안에 포함된 URL
  - 데이터: 피싱 URL(KISA 공개 데이터, 확보 완료) + 정상 URL(Tranco 상위 도메인 필터링 + 기존 공개 benign 데이터셋 보강)
  - 학습 방식: 2단계 접근
    1. 1단계 (baseline): URL lexical feature(길이, 특수문자 수, IP 직접 사용 여부, 서브도메인 개수, 유사도메인 여부 등) 추출 → XGBoost/RandomForest로 학습
    2. 2단계 (확장): URL 문자열을 문자 시퀀스로 보고 CNN/소형 Transformer 등 character-level 딥러닝 모델 학습
    3. 두 모델의 성능(정확도, F1, Recall 등)을 비교 분석해 정리
- **문구 설명 생성 (LLM 프롬프팅)**
  - URL 판별 결과 + 문자 원문을 LLM에 전달해, 왜 위험한지 + 대응 방법을 자연어로 설명
  - 별도 파인튜닝 없이 프롬프트 설계로 처리
  - 참고 자료: KISA 보호나라, 경찰청 사이버안전지킴이, 금융감독원 보이스피싱 사례집의 실제 문구 예시 (부족하면 알려진 사기 패턴 - 택배 사칭, 대출 사기, 지인 사칭 등을 모사한 합성 예시로 보강, 합성 데이터임을 명시)
- **데모 앱**
  - 사용자가 문자 메시지 전체를 텍스트로 붙여넣기
  - 시스템이 URL을 추출해 판별기로 위험도 판단
  - LLM이 판정 근거 + 대응 가이드를 함께 생성해 화면에 출력

### 제외

- 실시간 SMS 가로채기/차단 등 실제 서비스화 기능
- 모바일 앱 배포, 실사용자 대상 운영
- 문구 자체를 별도로 파인튜닝하는 분류기 (LLM 프롬프팅으로 대체)

## 제약

- 기한 없음 — 완성도 위주로 진행
- 데이터셋은 가능한 한 한국 데이터 우선 (단, URL 구조적 특징은 언어 종속적이지 않아 정상 URL 확보 시 해외 데이터 활용 가능)

## 완료 기준

- URL 분류기가 정량적 성능 지표(Recall 우선, 피싱을 놓치지 않는 것이 중요)로 평가됨
- baseline(classical ML) vs deep model 성능 비교 결과가 정리됨
- 데모 앱이 end-to-end로 동작 (문자 입력 → URL 추출/판별 → LLM 설명 생성 → 결과 출력)

## 열린 질문

- 정상 URL 확보 시 Tranco 필터링만으로 충분한 양/품질이 나올지 실제 확인 필요
- 문구 사례 자료(KISA/경찰청/금감원)의 실제 접근 가능 여부 및 저작권/이용 조건 확인 필요
- URL 분류기 정량 목표치(예: Recall 몇 % 이상)는 baseline 결과를 본 뒤 구체화
